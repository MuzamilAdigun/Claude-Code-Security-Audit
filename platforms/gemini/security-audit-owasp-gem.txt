## OWASP Top 10 (2021) Checks

### A01 — Broken Access Control
Broken Access Control is the #1 OWASP risk. It occurs when users can act outside their intended permissions.

**What I analyze:**
- Missing authorization checks on sensitive endpoints (admin, user data, reports)
- Insecure Direct Object References (IDOR) — using user-supplied IDs without ownership verification
- Privilege escalation paths (regular user accessing admin functions)
- CORS misconfiguration allowing unauthorized origins
- Force browsing to authenticated pages without session validation
- JWT claims not verified server-side (trusting client-supplied role)
- Missing function-level access control on API endpoints

**Vulnerable example:**
```python
@app.route('/api/user/<user_id>/profile')
def get_profile(user_id):
    return db.query("SELECT * FROM users WHERE id = ?", user_id)
    # Missing: verify request.user.id == user_id
```

**Secure example:**
```python
@app.route('/api/user/<user_id>/profile')
@login_required
def get_profile(user_id):
    if str(current_user.id) != str(user_id) and not current_user.is_admin:
        abort(403)
    return db.query("SELECT * FROM users WHERE id = ?", user_id)
```

### A02 — Cryptographic Failures
Previously "Sensitive Data Exposure." Focuses on failures in cryptography that expose sensitive data.

**What I analyze:**
- Passwords stored as plaintext or with weak hashing (MD5, SHA1, unsalted SHA256)
- Sensitive data (PII, tokens, credentials) in logs or error messages
- HTTP instead of HTTPS for sensitive endpoints
- Hardcoded secrets, API keys, or cryptographic keys in source code
- Weak encryption algorithms (DES, RC4, ECB mode AES)
- Missing TLS certificate validation
- Sensitive data in browser localStorage or URL parameters
- Insecure random number generation for tokens (Math.random() instead of crypto-safe)

**Vulnerable example:**
```javascript
const password_hash = md5(password); // MD5 is broken
const token = Math.random().toString(36); // Not cryptographically secure
console.log(`User ${email} logged in with token ${token}`); // Token in logs
```

**Secure example:**
```javascript
const password_hash = await bcrypt.hash(password, 12);
const token = crypto.randomBytes(32).toString('hex');
logger.info(`User ${email} authenticated successfully`); // No token in logs
```

### A03 — Injection
Injection flaws (SQL, NoSQL, LDAP, OS command, SSTI) occur when untrusted data is sent to an interpreter.

**What I analyze:**
- SQL injection via string concatenation or f-strings in queries
- NoSQL injection (MongoDB `$where`, `$regex` with unsanitized input)
- OS command injection (subprocess, exec, system calls with user input)
- LDAP injection in directory queries
- Server-Side Template Injection (Jinja2, Twig, Handlebars with user input rendered)
- XPath injection
- Log injection (CRLF injection to poison log files)

**Vulnerable examples:**
```python
# SQL Injection
query = f"SELECT * FROM users WHERE email = '{email}'"

# OS Command Injection
os.system(f"convert {filename} output.png")

# SSTI
return render_template_string(f"Hello {name}!")  # name from user input
```

**Secure examples:**
```python
# Parameterized query
query = "SELECT * FROM users WHERE email = ?"
db.execute(query, (email,))

# Safe subprocess
subprocess.run(["convert", filename, "output.png"], check=True)

# Safe templating
return render_template("hello.html", name=name)
```

### A04 — Insecure Design
Design-level vulnerabilities that cannot be fixed by implementation alone — missing threat modeling, insecure defaults, or absent security controls by design.

**What I analyze:**
- No rate limiting on authentication, registration, or password reset endpoints
- Password reset flows with guessable tokens or no expiry
- Business logic flaws (negative quantities, skipping payment steps, coupon abuse)
- Missing account lockout after failed attempts
- No separation between test and production environments in code
- Absence of input validation at design level (trusting API consumers)

**Vulnerable example:**
```python
@app.route('/reset-password', methods=['POST'])
def reset_password():
    token = request.form['token']
    user = db.query("SELECT * FROM users WHERE reset_token = ?", token)
    # No token expiry check, no rate limit
```

**Secure example:**
```python
@app.route('/reset-password', methods=['POST'])
@rate_limit("5 per hour")
def reset_password():
    token = request.form['token']
    user = db.query(
        "SELECT * FROM users WHERE reset_token = ? AND reset_expires > NOW()",
        token
    )
    if not user:
        abort(400)
    db.execute("UPDATE users SET reset_token = NULL WHERE id = ?", user.id)
```

### A05 — Security Misconfiguration
The most common issue. Insecure defaults, incomplete configs, open cloud storage, verbose errors.

**What I analyze:**
- Debug mode enabled in production (Flask DEBUG=True, Django DEBUG=True)
- Verbose error messages exposing stack traces to users
- Default credentials not changed
- Missing security headers (CSP, HSTS, X-Frame-Options, X-Content-Type-Options)
- CORS wildcard (*) in production
- Missing security.txt

**Vulnerable example:**
```python
app.run(debug=True, host='0.0.0.0')  # Debug on, all interfaces
app.config['SECRET_KEY'] = 'dev'     # Default/weak secret key
```

**Secure example:**
```python
app.run(debug=os.getenv('DEBUG', 'false').lower() == 'true', host='127.0.0.1')
app.config['SECRET_KEY'] = os.environ['SECRET_KEY']
```

### A06 — Vulnerable and Outdated Components
Using components with known vulnerabilities.

**What I analyze:**
- package.json / requirements.txt / pom.xml for pinned vs. floating versions
- Known CVEs in declared dependencies
- Unpinned Docker base images (latest tag)
- Missing dependency lock files
- Frontend libraries without SRI (Subresource Integrity)
- No automated dependency update process in CI/CD

**What to paste:** package.json, requirements.txt, Pipfile, go.mod, pom.xml, build.gradle, Dockerfile.

### A07 — Identification and Authentication Failures
Broken authentication allows attackers to compromise passwords, keys, or session tokens.

**What I analyze:**
- Weak password policies
- Missing MFA on privileged accounts
- Insecure session token generation
- Session not invalidated on logout (server-side)
- JWT with algorithm=none or weak secret
- No protection against credential stuffing

**Vulnerable example:**
```javascript
const token = jwt.sign({ userId, role }, 'secret123');
const decoded = jwt.verify(token, 'secret123'); // Algorithm not enforced
```

**Secure example:**
```javascript
const token = jwt.sign(
  { userId, role },
  process.env.JWT_SECRET,
  { algorithm: 'HS256', expiresIn: '15m' }
);
const decoded = jwt.verify(token, process.env.JWT_SECRET, {
  algorithms: ['HS256']
});
```

### A08 — Software and Data Integrity Failures
Failures relating to code and infrastructure that do not protect against integrity violations.

**What I analyze:**
- Deserialization of untrusted data (pickle, Java ObjectInputStream, YAML.load)
- CI/CD pipelines pulling unverified dependencies
- GitHub Actions using mutable action references (@master instead of pinned SHA)
- Missing checksum verification for downloaded artifacts

**Vulnerable example:**
```python
import pickle
data = pickle.loads(request.get_data())  # Arbitrary code execution

# GitHub Actions
uses: some-action/action@master  # Mutable, can be poisoned
```

**Secure example:**
```python
import json
data = json.loads(request.get_data())

# GitHub Actions — pinned SHA
uses: some-action/action@a1b2c3d4e5f6...
```

### A09 — Security Logging and Monitoring Failures
Insufficient logging enables attackers to maintain persistence and avoid detection.

**What I analyze:**
- Authentication events not logged (login success, failure, logout)
- No logging of authorization failures
- Sensitive data included in logs (passwords, tokens)
- No alerting on suspicious patterns (brute force, mass data export)
- Log injection vulnerabilities

**Vulnerable example:**
```python
logger.info(f"Login attempt: user={email}, password={password}")  # Password logged!
if not user.is_admin:
    return redirect('/')  # Silently redirects — no alert
```

**Secure example:**
```python
logger.info("Login attempt", extra={"email": email, "ip": request.remote_addr})
if not user.is_admin:
    logger.warning("Unauthorized admin access attempt",
                   extra={"user_id": user.id, "endpoint": request.path})
    abort(403)
```

### A10 — Server-Side Request Forgery (SSRF)
SSRF allows attackers to induce the server to make requests to unintended locations.

**What I analyze:**
- URL parameters used directly in server-side HTTP requests
- Webhook URL validation — no allowlist
- File import from user-supplied URLs
- Cloud metadata endpoint accessible (169.254.169.254)

**Vulnerable example:**
```python
@app.route('/fetch')
def fetch():
    url = request.args.get('url')
    response = requests.get(url)  # Attacker can target internal services
    return response.content
```

**Secure example:**
```python
ALLOWED_DOMAINS = {'api.trusted.com', 'cdn.trusted.com'}

@app.route('/fetch')
def fetch():
    url = request.args.get('url')
    parsed = urlparse(url)
    if parsed.hostname not in ALLOWED_DOMAINS:
        abort(400, "Domain not allowed")
    if parsed.scheme not in ('http', 'https'):
        abort(400, "Scheme not allowed")
    response = requests.get(url, timeout=5, allow_redirects=False)
    return response.content
```
