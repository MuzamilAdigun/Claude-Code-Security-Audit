## GDPR Requirements Checks

### Article 5 — Principles of Processing
The six core principles: lawfulness, fairness, transparency; purpose limitation; data minimization; accuracy; storage limitation; integrity and confidentiality.

**What I analyze:**
- Data minimization: Are unnecessary fields collected? (collecting date of birth when age verification would suffice)
- Purpose limitation: Is data used only for declared purposes? (email collected for registration later used for marketing without consent)
- Storage limitation: Are retention periods defined and enforced in code?
- Accuracy: Are there mechanisms to update/correct personal data?
- Data collected but never used (dead code with PII collection)

**Vulnerable example:**
```javascript
// Collecting excessive data — age, gender, phone not needed for newsletter signup
const signupSchema = {
  email: required,
  name: required,
  age: required,
  gender: required,
  phone: required,
  address: required
};
```

**Compliant example:**
```javascript
// Data minimization — only what is necessary
const signupSchema = {
  email: required,
  name: optional  // Only if used for personalization
};
```

---

### Articles 6 & 7 — Lawful Basis and Consent

**What I analyze:**
- Is a lawful basis documented for each processing activity?
- Is consent collected before processing? (especially for analytics, marketing cookies)
- Is consent granular, specific, and freely given?
- Is there a mechanism to withdraw consent?
- Are consent records stored with timestamp, version, and scope?
- Does "pre-ticked" consent exist (prohibited)?
- Is legitimate interest relied upon for tracking without proper Legitimate Interest Assessment (LIA)?

**Vulnerable example:**
```javascript
// Analytics loaded without consent
useEffect(() => {
  window.dataLayer = window.dataLayer || [];
  gtag('config', 'GA-XXXXXXXXXX');  // Tracking without consent
}, []);

// Pre-ticked marketing checkbox
<input type="checkbox" name="marketing" checked={true} />
```

**Compliant example:**
```javascript
// Load analytics only after consent
const { consent } = useCookieConsent();
useEffect(() => {
  if (consent.analytics) {
    window.dataLayer = window.dataLayer || [];
    gtag('config', 'GA-XXXXXXXXXX');
  }
}, [consent.analytics]);

// Unchecked by default
<input type="checkbox" name="marketing" defaultChecked={false} />
```

### Articles 12–14 — Transparency and Information Obligations

**What I analyze:**
- Privacy policy exists and is accessible from every page (footer link)
- Privacy policy covers: identity of controller, purposes, legal bases, retention periods, data subject rights, third-party sharing, international transfers
- Cookie notice appears before non-essential cookies are set
- Privacy policy language is plain and understandable (not legal jargon only)
- "Just-in-time" notices at data collection points (form descriptions, registration page)
- Last updated date on privacy policy

**Gap example:** Registration form collects phone number with no explanation of why it is needed, how long it is kept, or what third parties receive it.

---

### Articles 15–22 — Data Subject Rights

**What I analyze:**
- **Article 15 (Access):** Is there an endpoint to export all user data?
- **Article 17 (Erasure/Right to be Forgotten):** Is there a deletion endpoint? Does it cascade to all data stores (database, backups, logs, third-party systems)?
- **Article 18 (Restriction):** Can processing be restricted without deleting the account?
- **Article 20 (Portability):** Can users export data in machine-readable format (JSON, CSV)?
- **Article 21 (Objection):** Can users opt out of legitimate interest processing?
- SLA for responding to rights requests (maximum 30 days under GDPR)

**Vulnerable example:**
```python
@app.route('/delete-account', methods=['POST'])
def delete_account():
    db.execute("DELETE FROM users WHERE id = ?", current_user.id)
    # Missing: delete from analytics, email lists, backups, logs, CDN cache
```

**Compliant example:**
```python
@app.route('/delete-account', methods=['POST'])
@login_required
def delete_account():
    user_id = current_user.id
    email = current_user.email

    # Database deletion (cascade to related tables)
    db.execute("DELETE FROM users WHERE id = ?", user_id)

    # Remove from third-party systems
    mailchimp.delete_subscriber(email)
    analytics.delete_user(user_id)

    # Schedule backup anonymization
    erasure_queue.enqueue('anonymize_backups', user_id=user_id)

    # Log the erasure (retain the fact of deletion, not the data)
    audit_log.info("Account deleted", extra={"action": "erasure", "requester": user_id})

    return jsonify({"status": "deleted", "confirmation_id": generate_id()})
```

### Article 25 — Privacy by Design and by Default

**What I analyze:**
- Default settings are the most privacy-protective (opt-in, not opt-out)
- Pseudonymization used where possible (storing user_id instead of name in logs)
- Data minimization by design — collecting minimum fields
- Anonymization of analytics data where full tracking is not needed
- Separation of identifying data from behavioral data

**Gap example:** User activity logs contain full email addresses and names instead of pseudonymous IDs. Profile visibility defaults to "public" instead of "private."

---

### Article 30 — Records of Processing Activities (ROPA)

**What I analyze:**
- Evidence of a data inventory or ROPA (even as a spreadsheet referenced in documentation)
- All processing activities have documented: purpose, data categories, recipients, retention period, safeguards
- Third-party data processors listed (cloud providers, analytics, payment processors)

**Note:** The ROPA itself is a document, but code should document data flows via comments, data model annotations, or README sections.

---

### Article 32 — Security of Processing

**What I analyze:**
- Encryption at rest for personal data stores (database encryption, encrypted backups)
- Encryption in transit (TLS 1.2+ enforced, HSTS header)
- Password hashing with strong algorithms (bcrypt, Argon2, PBKDF2)
- Access controls — only authorized personnel access personal data
- Pseudonymization of user identifiers in logs and analytics
- Security testing evidence (penetration tests, vulnerability scanning)
- Backup encryption and restoration testing

**Vulnerable example:**
```python
# PII stored in plaintext in logs — Article 32 violation
logger.info(f"User registered: name={name}, email={email}, ip={request.remote_addr}, dob={date_of_birth}")
```

**Compliant example:**
```python
# Pseudonymous logging — only user_id
logger.info("User registered", extra={"user_id": new_user.id, "source": "web"})
```

---

### Articles 33 & 34 — Breach Notification

**What I analyze:**
- Incident response procedure exists with 72-hour DPA notification SLA
- Breach detection mechanisms in place (anomaly detection, alerting)
- Internal escalation path for security incidents (defined in documentation)
- Customer notification templates prepared

---

### Article 35 — Data Protection Impact Assessment (DPIA)

**What I analyze:**
- DPIA conducted (or evidence of assessment) for high-risk processing:
  - Large-scale processing of special category data
  - Systematic profiling
  - Processing of children's data
  - Biometric identification
  - AI-driven decision making with legal effect

---

### Articles 44–49 — International Data Transfers

**What I analyze:**
- Third-party services used that transfer data outside the EU (identified from import statements, API calls, environment variables referencing non-EU providers)
- Standard Contractual Clauses (SCCs) or adequacy decisions relied upon
- US-based services: AWS (us-east), Google, Meta APIs — transfer mechanism verified?
- Data residency configuration in cloud services (eu-west regions used?)

**Gap example:** Application connects to `us-east-1.rds.amazonaws.com` (US region) for EU user data without a Transfer Impact Assessment or SCCs in place.

---

### Cookie and Tracking Compliance

**What I analyze:**
- Essential cookies only before consent (no analytics, advertising, or personalization cookies pre-consent)
- Cookie banner implements genuine choice (accept/decline/customize)
- Consent Management Platform (CMP) integrated correctly
- Third-party scripts blocked until relevant consent given
- Cookie lifetime does not exceed declared retention period

**Vulnerable example:**
```html
<!-- Loads Meta Pixel before consent — illegal under GDPR/ePrivacy -->
<script>
  !function(f,b,e,v,n,t,s) { /* Meta Pixel */ }
  fbq('init', '1234567890');
  fbq('track', 'PageView');
</script>
```
