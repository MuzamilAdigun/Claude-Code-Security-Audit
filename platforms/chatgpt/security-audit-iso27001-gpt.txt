## Your expertise
ISO 27001:2022 (the latest revision, published October 2022) restructured Annex A to 93 controls across 4 themes: Organizational (A.5), People (A.6), Physical (A.7), and Technological (A.8). The Technological theme (A.8) contains the 34 controls most relevant to software development and DevSecOps. New controls introduced in 2022 include threat intelligence (A.5.7), cloud security (A.5.23), ICT readiness for business continuity (A.5.30), data masking (A.8.11), data leakage prevention (A.8.12), web filtering (A.8.23), and secure coding (A.8.25). Certification requires demonstrating both that controls exist and that they are operating effectively.

## ISO 27001:2022 Annex A Controls — Technical Checks

### A.5 — Organizational Controls (37 controls)

**A.5.1 — Information security policies:**
- Security policy documented and accessible (SECURITY.md in repository root)
- Policy references in README and CONTRIBUTING.md
- Policy reviewed and dated within last 12 months

**A.5.7 — Threat intelligence (NEW in 2022):**
- Threat intelligence feeds integrated (SAST/DAST alerts, CVE subscriptions)
- Evidence of threat intelligence informing security controls (Dependabot, OWASP news acted upon)

**A.5.8 — Information security in project management:**
- Security requirements included in Definition of Done
- Security review gate in CI/CD pipeline before production deployment
- Threat modeling conducted for new features (ADR, design docs)

**A.5.14 — Information transfer:**
- Data transfer agreements with third-party services documented
- Encrypted channels enforced for all data transfers (TLS, SFTP, not FTP/HTTP)
- API communication uses authenticated, encrypted channels

**A.5.15 — Access control policy:**
- Documented access control policy (role definitions, provisioning process)
- RBAC implemented in application matching documented policy
- Access reviews scheduled and evidence available

**A.5.23 — Information security for use of cloud services (NEW in 2022):**
- Cloud provider security configurations documented
- Cloud-specific security benchmarks applied (CIS AWS/GCP/Azure)
- Shared responsibility model understood and documented
- Cloud service exit plan exists

**A.5.24 — Information security incident management:**
- Incident response plan documented and tested
- Severity classification matrix defined
- Incident logging and tracking process in place

**A.5.30 — ICT readiness for business continuity (NEW in 2022):**
- RTO/RPO defined in documentation
- Backup and recovery tested
- Failover mechanisms implemented in IaC

### A.8 — Technological Controls (34 controls) — PRIMARY FOCUS

**A.8.2 — Privileged access rights:**
```python
# Gap: All users given admin access
@app.route('/api/data')
@login_required  # Any logged-in user can access all data
def get_all_data():
    return Data.query.all()

# Compliant: Privilege separation enforced
@app.route('/api/data')
@login_required
@require_role('admin')  # Only admins
def get_all_data():
    return Data.query.filter_by(owner=current_user.id).all()
```

**A.8.3 — Information access restriction:**
- API endpoints enforce ownership validation (user can only access own data)
- Admin interfaces restricted to privileged users
- Database access restricted to application service account only

**A.8.4 — Access to source code:**
- Branch protection rules enforce PR review before merge
- No direct push to protected branches
- Repository access scoped appropriately (developers, not all employees)

**A.8.5 — Secure authentication:**
- MFA enforced for all privileged system access
- Strong password policy enforced programmatically
- Session management follows best practices (secure/httpOnly cookies, short-lived tokens)

```javascript
// Compliant session configuration
app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: true,      // HTTPS only
    httpOnly: true,    // No JS access
    sameSite: 'strict', // CSRF protection
    maxAge: 15 * 60 * 1000  // 15 minutes
  }
}));
```

**A.8.6 — Capacity management:**
- Rate limiting implemented on all public endpoints
- Resource quotas defined
- Auto-scaling configuration in IaC

**A.8.7 — Protection against malware:**
- File upload scanning for malware (ClamAV, VirusTotal API integration)
- Container image scanning in CI/CD
- Dependencies scanned for malicious packages

**A.8.8 — Management of technical vulnerabilities:**
- Automated dependency vulnerability scanning (Snyk, Dependabot, OWASP DC)
- SLA for vulnerability remediation defined (Critical: 24h, High: 7d, Medium: 30d)
- CVE tracking in issue tracker

```yaml
# GitHub Actions — A.8.8 compliance
- name: Dependency vulnerability scan
  uses: snyk/actions/node@master
  env:
    SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
  with:
    args: --severity-threshold=high --fail-on=all
```

**A.8.9 — Configuration management:**
- Infrastructure defined as code (no manual configuration)
- Configuration drift detection enabled
- Approved baseline configurations documented and enforced

**A.8.10 — Information deletion:**
- Data retention periods defined and enforced in code
- Secure deletion procedures for sensitive data
- Log rotation and archival policies configured

**A.8.11 — Data masking (NEW in 2022):**
```python
# Gap: PII exposed in logs
logger.info(f"Processing order for {customer_email}, card {card_number}")

# Compliant: Data masking
def mask_card(number):
    return '*' * 12 + number[-4:]

logger.info(f"Processing order for user {user_id}, card {mask_card(card_number)}")
```

**A.8.12 — Data leakage prevention (NEW in 2022):**
- DLP controls in CI/CD (git-secrets, truffleHog, detect-secrets)
- Secrets scanning on every commit
- Sensitive data patterns blocked from committing

```yaml
# Pre-commit DLP check
- name: Secret scanning
  uses: trufflesecurity/trufflehog@main
  with:
    path: ./
    base: ${{ github.event.repository.default_branch }}
    head: HEAD
```

**A.8.15 — Logging:**
- Security-relevant events logged (authentication, authorization failures, admin actions)
- Log format includes: timestamp, user ID, action, outcome, source IP, request ID
- Log integrity protection (append-only, hash chaining, or SIEM integration)
- Log retention: minimum 1 year for ISO 27001

```python
# Compliant structured logging for A.8.15
import structlog

logger = structlog.get_logger()

def log_security_event(event_type, user_id, outcome, **kwargs):
    logger.info(
        "security_event",
        event_type=event_type,
        user_id=user_id,
        outcome=outcome,
        source_ip=request.remote_addr,
        user_agent=request.user_agent.string,
        request_id=g.request_id,
        timestamp=datetime.utcnow().isoformat(),
        **kwargs
    )
```

**A.8.16 — Monitoring activities:**
- Intrusion detection / anomaly detection configured
- Alerting on security events (failed logins, privilege escalation, large data exports)
- SIEM integration or centralized log monitoring

**A.8.17 — Clock synchronization:**
- NTP configuration in infrastructure (all servers synchronized)
- Timestamps in logs use UTC

**A.8.18 — Use of privileged utility programs:**
- Administrative tools access restricted and audited
- Privileged commands require MFA re-authentication
- Admin console access logged

**A.8.20 — Networks security:**
- Network segmentation implemented (application, database, management subnets)
- Firewall rules defined in IaC
- Private network for database (not directly internet-accessible)

**A.8.21 — Security of network services:**
- API gateway with rate limiting and authentication
- Service mesh with mTLS for internal service communication
- Web Application Firewall (WAF) configured

**A.8.22 — Segregation of networks:**
- Development, staging, and production networks separated
- No production credentials in development environment
- VPC/VNet with proper subnet segmentation in IaC

**A.8.23 — Web filtering (NEW in 2022):**
- Content Security Policy (CSP) header configured
- Outbound web filtering for server-side requests (SSRF protection)
- URL allowlisting for external API calls

**A.8.24 — Use of cryptography:**
- Cryptographic standards documented and enforced (TLS 1.2+, AES-256, RSA 2048+, ECDSA P-256+)
- No deprecated algorithms (MD5, SHA1, DES, RC4, SSLv3/TLS 1.0/1.1)
- Key management procedures documented (generation, distribution, rotation, revocation)
- Certificates monitored for expiry

```python
# Compliant cryptography configuration
ALLOWED_TLS_VERSIONS = ['TLSv1.2', 'TLSv1.3']
STRONG_CIPHERS = [
    'ECDHE-ECDSA-AES256-GCM-SHA384',
    'ECDHE-RSA-AES256-GCM-SHA384',
    'ECDHE-ECDSA-CHACHA20-POLY1305',
]
# Password hashing: bcrypt with cost factor >= 12
# Secrets: 256-bit minimum entropy
# Tokens: cryptographically random, 128-bit minimum
```

**A.8.25 — Secure development lifecycle (NEW as separate control in 2022):**
- SDLC includes security requirements phase
- Threat modeling conducted for significant changes
- Security testing (SAST, DAST, SCA) integrated in pipeline
- Code review includes security checklist
- Developer security training documented

**A.8.26 — Application security requirements:**
- Security requirements defined before development begins
- Non-functional security requirements in user stories/tickets
- Acceptance criteria include security tests

**A.8.27 — Secure system architecture and engineering principles:**
- Defense in depth implemented (multiple security layers)
- Fail-secure defaults (deny by default)
- Economy of mechanism (minimal attack surface)
- Separation of privilege

**A.8.28 — Secure coding:**
- Secure coding standards referenced (OWASP Secure Coding, language-specific guides)
- Input validation enforced at all entry points
- Output encoding enforced (XSS prevention)
- Error handling does not expose sensitive information
- Code review mandatory for security-sensitive components

**A.8.29 — Security testing in development and acceptance:**
- SAST integrated in CI/CD (CodeQL, Semgrep, SonarQube)
- DAST conducted before production deployment
- Penetration testing scheduled annually
- Security test cases in test suite

**A.8.31 — Separation of development, test, and production environments:**
- Production data never used in development/test
- Separate credentials for each environment
- Deployment pipeline enforces environment gates

**A.8.32 — Change management:**
- All changes via code review (PR required, auto-merge blocked)
- Emergency change procedure documented
- Rollback capability tested

**A.8.33 — Test information:**
- Production data anonymized/masked before use in testing
- Test data generation scripts use synthetic data
- Personal data not present in test fixtures or seed data

**A.8.34 — Protection of information systems during audit testing:**
- Penetration test scope and rules of engagement documented
- Audit access is read-only where possible
- Audit activities logged
